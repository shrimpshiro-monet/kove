## 2. Discovery: The Full Pipeline Runs Locally

The user requested 10 E2E test exports. Initial belief was that the render pipeline required Docker (based on `checkDockerRunning()` guard in `vibe-render.ts:97-104`). Investigation revealed a critical architectural detail:

### Render fallback chain (`scripts/monet_pipeline.py:840-931`)

```
render_with_editly(edl, output, music)
  ├── macOS → render_in_docker(edl, output, music)
  │     ├── Docker available → containerized render with gl-transitions
  │     └── Docker unavailable → render_native(edl, output, music)  ← FALLBACK
  └── Linux → render_native(edl, output, music)
```

On macOS, `render_with_editly` first checks `render_in_docker` which probes Docker availability. If Docker is missing or fails, it falls back to `render_native()` — which uses **ffmpeg directly** with zero container dependencies.

## 3. Bug Fix: Beat Type Mismatch in `footage_analyzer.py`

### Symptom

First smoke test crashed at `_rank_segments()` in `scripts/analyzers/footage_analyzer.py:316`:

```
TypeError: unsupported operand type(s) for -: 'int' and 'dict'
```

### Root cause analysis

The `detect_beats()` function in `scripts/analyzers/beat_detector.py:24-50` returns a dict with structure:

```python
{
    "tempo_bpm": 99.4,
    "beats": [{"time": 0.45, "strength": 0.92}, {"time": 1.05, "strength": 0.87}, ...],
    "beat_count": 30,
    "avg_beat_interval": 0.603,
    "beat_method": "librosa",
}
```

The `beats` value is a `List[Dict]`, not `List[float]`. However, `_rank_segments()` at line 316 does:

```python
min_dist = min(abs(seg["start"] - b) for b in beats)
```

This expects `b` to be a float, but gets a `{"time": ..., "strength": ...}` dict.

### Why the bug existed

In `scripts/analyzers/footage_analyzer.py:100-117`, there are two code paths for extracting beats:

**Path A — music file provided (line 100-101):**
```python
beat_result = detect_beats(audio_tmp)
beats = beat_result.get("beats", [])
# ← beats is List[Dict], passed directly to _rank_segments()
```

**Path B — footage's own audio (line 114-116):**
```python
beat_result = detect_beats(audio_tmp)
raw_beats = beat_result.get("beats", [])
beats = [b["time"] if isinstance(b, dict) else b for b in raw_beats]
# ← correctly converts to List[float]
```

Path B had the conversion. Path A did not. The previous developer fixed Path B but missed Path A.

### Fix applied

At `scripts/analyzers/footage_analyzer.py:100-102` (Path A):

```python
# BEFORE:
beat_result = detect_beats(audio_tmp)
beats = beat_result.get("beats", [])

# AFTER:
beat_result = detect_beats(audio_tmp)
raw_beats = beat_result.get("beats", [])
beats = [b["time"] if isinstance(b, dict) else b for b in raw_beats]
```

### Verification

Same smoke test re-run after fix. Step 2 (EDL generation) completed without crash. Full pipeline through render: 120s, 658KB output.

## 5. The "CRT and Colors" Problem

### What the user observed

The rendered exports had unnatural visual artifacts: desaturated colors, vignette darkening, blur effects, and overall "CRT monitor" appearance applied to clean gameplay and talking-head footage.

### Root cause

The `generate_edl_from_dna()` function in `scripts/monet_pipeline.py:379-458` applies **reference-derived effects** to each footage clip:

```python
# Line 978-992 in render_native():
for effect_type in unique_effects:
    if effect_type == "blur":
        vf_parts.append("boxblur=8:8")
    elif effect_type == "vignette":
        vf_parts.append("vignette=PI/4")
    elif effect_type == "flash":
        vf_parts.append("eq=brightness=0.3")
    elif effect_type == "shake":
        vf_parts.append("crop=w=in_w-10:h=in_h-10:x=5:y=5")
    elif effect_type == "glow":
        vf_parts.append("unsharp=5:5:1.5")
    elif effect_type == "desaturation":
        vf_parts.append("eq=saturation=0.3")
```

These effects are detected from the **reference videos** (which are Instagram/TikTok edits with heavy visual filters) and faithfully replicated onto the raw footage. The reference edits use desaturation, vignette, and blur as stylistic choices for their content — but applying these to game footage or vlog talking heads produces degraded, unnatural output.

Additionally, shot-type grading applies extra contrast for extreme close-ups:

```python
if shot_type == "extreme_close":
    vf_parts.append("eq=contrast=1.1")
```

### Fix strategy

In the v2 batch runner, effects are stripped from the EDL before rendering:

```python
for track in edl.get("timeline", {}).get("tracks", []):
    for clip in track.get("clips", []):
        clip["effects"] = []
        clip["visualEffects"] = []
```

This preserves the reference's editing **structure** (cut pacing, shot selection duration, narrative arc ordering) while removing the visual effect application that degrades output quality.

### Engine-by-engine detail

#### Engine 1: `get_video_info()` 
**Source**: `monet_pipeline.py` (helper)
**What it does**: Runs `ffprobe` to extract width, height, duration, fps, codec, has_audio flag, file size.
**Output**: `ref-video_info.json`
```json
{
  "width": 1920, "height": 1080, "duration": 12.34, "fps": 30.0,
  "codec": "h264", "has_audio": true, "size_mb": 8.0
}
```

#### Engine 2: `classify_reference_type()`
**Source**: `analyzers/reference_type_classifier.py`
**What it does**: Extracts 5 keyframes from the video, sends to LLM (via DigitalOcean Nemotron or heuristic fallback). Classifies as: `action`, `drama`, `sports`, `music_video`, `vlog`, `commercial`, `documentary`, `tutorial`, `gaming`, or `unknown`.
**Output**: `ref-reference_type.json`
```json
{
  "type": "sports",
  "confidence": 0.85,
  "description": "Fast-paced sports edit with quick cuts and motion"
}
```

#### Engine 3a: `detect_cuts()`
**Source**: `monet_pipeline.py` (via PySceneDetect or ffmpeg threshold method)
**What it does**: Analyzes frame-by-frame intensity differences to find scene changes. Uses `threshold=0.15` for sensitivity.
**Output**: `ref-cuts.json` — array of `{"time": float, "frame": int}` objects.

#### Engine 3b: Shot builder (inline in `extract_grammar`)
**What it does**: Converts cut timestamps into shot segments, filtering out sub-frame segments (<0.034s ≈ 1 frame at 30fps).
**Output**: `ref-shots.json` — array of `{"index": int, "start": float, "end": float, "duration": float}` objects.

#### Engine 4: `analyze_motion()` + `compute_motion_stats()`
**Source**: `analyzers/motion_analyzer.py`
**What it does**: 
1. Reads video frames at 10fps
2. Computes dense optical flow via Farneback algorithm (`cv2.calcOpticalFlowFarneback`)
3. Calculates per-frame motion magnitude and angle
4. Classifies camera motion (static, pan, tilt, zoom, handheld) and subject motion (low, medium, high)

**Output**: `ref-motion.json` — raw flow data + statistics
```json
{
  "data": [
    {"frame": 0, "magnitude": 0.15, "angle": 45.2, "camera": "static", "subject": "low"},
    ...
  ],
  "stats": {
    "avg_magnitude": 0.258,
    "max_magnitude": 2.1,
    "camera_motion_distribution": {"static": 0.6, "pan": 0.2, "handheld": 0.2},
    "subject_motion_distribution": {"low": 0.5, "medium": 0.3, "high": 0.2}
  }
}
```

#### Engine 5: `detect_beats()` + `analyze_rhythm()`
**Source**: `analyzers/beat_detector.py`
**What it does**:
1. Extracts audio via ffmpeg to 16-bit PCM WAV (44100Hz, mono)
2. Runs `librosa.load()` + `librosa.beat.beat_track()` for tempo and beat frames
3. Converts frames to timestamps, computes onset strength for each beat
4. `analyze_rhythm()` computes pattern analysis (downbeat likelihood, rhythmic complexity)

**Output**: `ref-beats.json`
```json
{
  "tempo_bpm": 99.4,
  "beat_count": 30,
  "avg_beat_interval": 0.603,
  "beat_method": "librosa",
  "beats": [
    {"time": 0.45, "strength": 0.92},
    {"time": 1.05, "strength": 0.87},
    ...
  ],
  "rhythm": {
    "cuts_on_beat": 0.73,
    "complexity": 0.45,
    "downbeat_likelihood": 0.6
  }
}
```

#### Engine 6: `analyze_color()`
**Source**: `analyzers/color_analyzer.py`
**What it does**:
1. Samples frames at 2fps via ffmpeg
2. Converts each frame to HSV
3. Runs k-means clustering (k=5) on pixel colors to find dominant palette
4. Classifies color grade as: `vibrant`, `desaturated`, `warm`, `cool`, `sepia`, `monochrome`, `hdr`, `pastel`, or `normal`

**Output**: `ref-color.json`
```json
{
  "grade": "desaturated",
  "dominant_colors": [
    {"h": 220, "s": 0.15, "v": 0.45, "hex": "#4a6b7c", "pct": 0.35},
    ...
  ],
  "avg_saturation": 0.22,
  "avg_brightness": 0.48,
  "contrast": 0.65,
  "warmth": -0.3
}
```

#### Engine 7: `classify_shot_type()` + `aggregate_shot_types()`
**Source**: `analyzers/shot_type_classifier.py`
**What it does**:
1. Extracts the middle frame from each shot
2. Runs a CNN classifier (ResNet50-based) for shot type
3. Classes: `extreme_wide`, `wide`, `medium`, `medium_close`, `close`, `extreme_close`
4. Aggregates distribution across all shots

**Output**: `ref-shot_types.json`
```json
{
  "per_shot": [
    {"shot_index": 0, "shotType": "wide", "confidence": 0.92},
    ...
  ],
  "summary": {

  }
```

#### Engine 8: `detect_effects()` + `aggregate_effects()`
**Source**: `analyzers/effect_detector.py`
**What it does**:
1. Analyzes each shot for visual effects by comparing frame differences and color histograms
2. Detects: `cut`, `fade`, `dissolve`, `wipe`, `blur`, `vignette`, `flash`, `shake`, `glow`, `desaturation`
3. Detects transitions between shots
4. `aggregate_effects()` counts totals and deduplicates

**Output**: `ref-effects.json`
```json
{
  "per_shot": [
    {"shot_index": 0, "effects": ["cut"], "transitions": [], "visualEffects": ["desaturation"]},
    ...
  ],
  "summary": {
    "totalEffects": 17,
    "effectDistribution": {"desaturation": 8, "vignette": 5, "flash": 4},
    "totalTransitions": 3,
    "transitionTypes": {"fade": 2, "dissolve": 1}
  }
}
```

#### Engine 9: `detect_text()` + `aggregate_text_results()`
**Source**: `analyzers/text_detector.py`
**What it does**:
1. Extracts frames at shot boundaries
2. Runs pytesseract OCR on each frame
3. Detects text regions, confidence, and approximate position

**Output**: `ref-text.json`
```json
{
  "per_shot": [
    {"shot_index": 0, "hasText": true, "textCount": 2, 
     "properties": {"positions": ["top", "bottom"], "avg_confidence": 0.85}}
  ],
  "summary": {
    "shots_with_text": 14,
    "total_text_regions": 28,
    "total_shots": 14
  }
}
```

#### Engine 10: `detect_speed_ramps()` + `aggregate_speed_results()`
**Source**: `analyzers/speed_ramp_detector.py`
**What it does**:
1. Analyzes frame-to-frame temporal differences
2. Detects speed changes (slow motion, fast forward, speed ramps)
3. Classifies each shot's speed: `normal`, `slow`, `fast`, `ramp_up`, `ramp_down`

**Output**: `ref-speed.json`
```json
{
  "per_shot": [
    {"shot_index": 0, "speed": "slow", "multiplier": 0.5, "confidence": 0.88},
    ...
  ],
  "summary": {
    "avgSpeed": 1.75,
    "speedDistribution": {"normal": 0.3, "slow": 0.4, "fast": 0.2, "ramp_up": 0.1}
  }
}
```

#### Engine 11: `analyze_semantic_events()` + `aggregate_semantic_results()`
**Source**: `analyzers/semantic_analyzer.py`
**What it does**:
1. Groups shots into batches of ~5
2. For each batch, sends extracted frames to LLM with a structured prompt asking for event classification
3. Falls back to heuristic (motion-based classification) if LLM fails
4. Event types: `establishing`, `setup`, `building`, `climax`, `reaction`, `transition`, `resolution`, `title_card`

**Output**: `ref-semantic.json`
```json
{
  "per_shot": [
    {"shot_index": 0, "event_type": "setup", "importance": 7, "description": "Opening scene establishing character"},
    ...
  ],
  "summary": {
    "event_counts": {"setup": 10, "reaction": 1, "transition": 3},
    "narrative_arc": "setup-heavy (71.4%)",
    "avg_importance": 6.2
  }
}
```
